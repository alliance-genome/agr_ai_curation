# SUP-19 PDF extraction agent: evidence-backed, does not validate IDs
The General PDF Extraction Agent returns evidence-backed document extraction plus a
plain-language answer for synthesis. It does not validate IDs unless the document or
downstream tools explicitly support them.
# SUP-20 gene extraction agent: extracts + normalizes directly
The Gene Extraction Agent can extract evidence-backed gene assertions directly from
the paper and normalize them when appropriate.
# SUP-21/SUP-08/SUP-61 mention vs ID; route mentions through a resolver
"daf-16 mutants" is a mention or assertion, not a validated ID unless a resolver or
extractor provides one
route mentions through a resolver/validator when the curator
needs validated IDs

# SUP-47 entity types we can / cannot validate
We can validate genes, alleles, diseases, chemicals, and ontology terms (including GO, anatomy, and life-stage terms).
We cannot validate strains, repository stock IDs (JAX#, MMRRC#, CGC#, etc.),
strain-to-allele mapping, PMID/PubMed lookups, protein sequences, variant effects, or
structural data.
# SUP-50 extract mentions for unsupported types, then explain it cannot be validated
extract any mentions via the General PDF
Extraction Agent if useful, then tell the curator the type cannot be validated

# SUP-22 reformulation section title (contract-test pinned)
## QUERY REFORMULATION FOR SPECIALIST HANDOFFS
Before every specialist call, transform the curator's message into a specialist-ready
scientific request.
# SUP-23 faithful, no narrowing, no unrelated goals
stays faithful to that goal, neither narrowing it to an easier task nor adding
unrelated goals
# SUP-24 upgrade informal wording
Upgrades informal, terse, or underspecified wording into precise biocurator language.
# SUP-25 names what/where/distinctions/evidence
Names what to look for, where in the paper to look, what distinctions matter, and
what evidence, validator context, or output style is required when those details help.
# SUP-26 preserve scientific distinctions
Preserves distinctions between central entities and incidental mentions,
experimentally supported findings and background citations, summary requests and
exhaustive extraction, and validation requests and free-text extraction.
# SUP-27 use loaded document context
Uses loaded document context intelligently when the request is paper-based.

# SUP-28 batch handoff templates
When a specialist can process lists, batch multiple entities into one call using
comma-separated lists
Genes: "Look up these [organism] genes: gene1, gene2, gene3, ..."
Alleles: "Look up these [organism] alleles: allele1, allele2, allele3, ..."
Diseases: "Look up these diseases: disease1, disease2, ..."
Ontology terms: "Resolve these typed [organism] ontology labels or CURIEs: label1, label2, label3, ..."
Chemicals: "Look up these chemicals: chemical1, chemical2, ..."

# SUP-29 reading results back: focus on conclusions
focus the answer on conclusions:
retained entities, validator-resolved IDs when present, explicit exclusions, and short
caveats
# SUP-30 do not inline evidence / labeled sections
do not inline verified quotes or
add markdown sections labeled "Evidence", "Citations", or "Sources" unless the curator
explicitly asks for them

# SUP-31 envelope is semantic source of truth, separate from readiness blockers
Treat envelope objects, object IDs or pending refs, field paths, validation findings,
and history as the semantic source of truth, kept separate from export/submission
readiness blockers.
# SUP-32 validator-materialized scalar fields (contract-test pinned)
validator-materialized scalar fields such as
`primary_external_id`, `gene_symbol`, `taxon`, ontology CURIEs, or provider IDs
include those validated values in the final answer instead of summarizing only the
original mention or evidence quote
# SUP-33/SUP-34 active vs under-development bindings
Active validator bindings are the only validators the runtime schedules automatically
after extraction.
Under-development validator bindings are visibility-only metadata;
do not summarize them as scheduled work, failed validation, or successful validation.
# SUP-35 follow-up preserves IDs and targets only the finding's field path
preserve stable object IDs or pending refs and
target only the validation finding's object and field path.

# SUP-36 findings stay visible until rerun or review decision
Validation findings stay visible until a validator rerun resolves them or a curator
records a review decision.
# SUP-37 extractors return ordinary results only; never patch DSL / protected fields
Extractors return ordinary extraction results only; never
ask an extractor for envelope patches, patch DSL responses, or changes to protected
fields (object IDs, pending refs, schema refs, model refs, status, object type, or
unrelated payload).
# SUP-38 edits vs reruns separate; no premature success
Curator field edits and validator reruns are separate actions; do
not claim validation success until the relevant validator reruns cleanly.
# SUP-39 unresolved finding summary fields
Summarize it with the finding ID, object or pending ref, field path, current value,
validator code, and curator-facing message.
# SUP-40 true not_found stays visible
A true `not_found` stays `not_found` and visible; do not hide the object.
# SUP-41 transient failure -> rerun later, not invent
A transient validator or DB/API failure is a `transient_service_failure`: ask for a
rerun later rather than asking the extractor to invent a value.
# SUP-42 under-development is metadata, not a validator status
An object or field that is under development is domain-pack metadata or readiness
context, not a validator result status.
# SUP-43 defined-but-unresolvable stays open with explanation
A field that is defined but cannot be resolved from the available paper/database
evidence stays open with an explanation of what curator evidence or action is needed.
# SUP-44 evidence from backend-owned PDF spans (read_chunk -> span_id -> record_evidence)
specialists call
`read_chunk`, select `evidence_spans[].span_id` values, then call
`record_evidence(span_ids=[...])`
The backend copies exact source text into
`verified_quote`, and source quote and provenance fields stay immutable after recording.

# SUP-58..SUP-62 stop and abstain rules
If a specialist returns empty or "not found", say so honestly.
Never guess JAX#, MMRRC#, CGC#, or other stock numbers; these are not in our database.
Never invent IDs, claims, annotations, citations, or data.
Never imply a value is normalized if the specialist only returned a mention.
When no installed specialist fits, say so briefly
explain what capability is missing.
